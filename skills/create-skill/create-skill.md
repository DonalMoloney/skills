# Create Skill

## The Law
**Every skill begins with one non-negotiable rule stated as an imperative command. This is "The Law" — iron rules that the skill enforces, not guidelines.**

## When to Use
- You are creating a new skill for your personal workflow
- A workflow pattern repeats often enough to codify
- A repeated mistake suggests a skill that prevents it
- **Never skip when:** the skill is for widespread use — invest in clarity

## Process

### Phase 1: Identify the Law
1. What is the non-negotiable rule this skill enforces?
2. State it as an imperative command, not a suggestion
3. Examples:
   - "Every commit must have a purpose statement"
   - "Never ship code without testing the happy path"
   - "Do not mark work complete without verification"

### Phase 2: Define When to Use
1. When should this skill be invoked?
2. What scenarios are common triggers?
3. What scenarios should NOT use this skill?
4. When would someone rationalize skipping it?

### Phase 3: Design the Process
1. Break the skill into 5-7 phases
2. Each phase has 1-3 steps (specific, actionable)
3. Include examples where helpful
4. Keep each phase to one clear purpose

### Phase 4: Identify Red Flags
1. What are the stop-immediately conditions?
2. What would indicate the skill is being misused?
3. What mistakes would this skill prevent?
4. List 5-8 red flags

### Phase 5: Add Common Rationalizations
1. What excuses do people make to skip the skill?
2. Why is each excuse wrong?
3. Create a table: Excuse | Why It's Wrong
4. Include 5-8 rationalizations

### Phase 6: Add Quick Reference
1. Summarize each phase in one row
2. Include: Phase name, Core action, Done when
3. Make it scannable

### Phase 7: Format and Deliver
1. Use the uniform template with frontmatter
2. Use clear formatting: **bold** for emphasis, code blocks for examples
3. Keep each section concise (max 300 words)
4. Proofread and test the skill

## Red Flags — Stop Immediately

- The skill has no clear "Law" — define it first
- The process is more than 7 phases — you're trying to do too much
- The skill has no red flags — you haven't thought about failure modes
- The skill contradicts existing skills — coordinate or combine them
- The skill is longer than 1500 words — trim or split it

## Common Rationalizations

| Excuse | Why It's Wrong |
|--------|---|
| "The skill should be flexible" | Flexible is vague. Iron laws are actually freeing — they reduce decision fatigue. |
| "We don't need a skill for this, it's obvious" | If it's obvious, people won't skip it. Skills codify non-obvious patterns. |
| "This skill is too much overhead for the benefit" | A skill you use 10 times saves 10× the creation cost. Amortize carefully. |
| "We can add more phases later" | You can, but it makes the skill harder to follow. Design completely before launching. |

## Quick Reference

| Phase | Core Action | Done When |
|-------|---|---|
| **Law** | State one non-negotiable rule as imperative | Rule is clear and unambiguous |
| **When to Use** | List triggers and non-triggers | Skill knows when it applies |
| **Process** | Design 5-7 phases with specific steps | Each phase is actionable |
| **Red Flags** | List stop-immediately conditions | 5-8 red flags are explicit |
| **Rationalizations** | Excuse table with counter-arguments | 5-8 rationalizations addressed |
| **Reference** | Summarize each phase in one row | Quick scan shows full flow |
| **Format** | Use template; check spelling; test | Skill is ready to use |
